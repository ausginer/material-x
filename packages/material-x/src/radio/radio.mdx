## When to use

- Use when users must select exactly one option from a set of two or more mutually exclusive choices.
- Use when all options need to be visible at the same time for comparison.

## When not to use

- Multiple independent selections. Use `mx-checkbox` instead.
- A single on/off choice. Use `mx-checkbox` or `mx-switch-button` instead.
- More than ~6 options. Consider a select/dropdown to save space.

## Grouping

Radio buttons of a group share the same `name` attribute, which is the key their `value` is submitted under.

`name` does **not** enforce mutual exclusion. Each `mx-radio` renders its own `<input type="radio">` inside its own shadow root, so native radio grouping does not apply — same-named radios can all be checked at once. As a controlled component, `mx-radio` leaves single selection to the host application: listen for `change` and set `checked` on exactly one radio, as the example below does.

Arrow-key navigation and `role="radiogroup"` semantics are likewise not provided; see [Accessibility](#accessibility).

## Accessibility

- **Keyboard:** `Tab` moves focus to each radio in turn; `Space` selects the focused radio. Arrow-key roving **is not implemented** — every radio is a separate tab stop, which departs from the expected radio-group keyboard pattern.
- **Focus indicator:** Drawn around the radio icon (not the full touch target).
- **Name:** Provide an accessible name via a wrapping `<label>` or `aria-label`.
- **Grouping:** `role="radiogroup"` is **not** provided automatically. Wrap a group in an element with `role="radiogroup"` and give it an accessible name, or assistive technology will announce each radio in isolation.
- **Disabled:** `disabled` blocks interaction and form participation.

> **Known gap.** Because roving tabindex and `radiogroup` semantics are absent, a bare set of `mx-radio` elements does not meet the expected radio-group interaction pattern. Until a grouping component exists, the host application must supply the wrapper role, the accessible name, and single-selection logic.

## Content guidelines

- List options in a logical order (alphabetical, chronological, or by likelihood).
- Keep labels short and mutually exclusive in meaning.
- Use sentence case for labels.

## Material 3 reference and deltas

- Material 3 overview: [https://m3.material.io/components/radio-button/overview](https://m3.material.io/components/radio-button/overview)
- Material 3 guidelines: [https://m3.material.io/components/radio-button/guidelines](https://m3.material.io/components/radio-button/guidelines)
- Deltas:
  - This is a controlled component — the host application manages `checked` state in response to `change`/`input` events.
  - There is no group wrapper component, and native browser grouping does not apply across shadow roots. Single selection, arrow-key roving, and `role="radiogroup"` are therefore the host application's responsibility.
